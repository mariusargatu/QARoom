# The incident → claim ritual

A postmortem that ends in "we'll be more careful" decays by the next sprint. The QARoom feedback loop ends
differently: **an incident yields a falsifiable claim card** — a manifest entry, a deliberate-bug toggle,
and a gate that goes RED when the bug is armed. The lesson stops being a memory and becomes a check that
fails the build if the failure mode ever returns.

This is the operating ritual behind [ADR-0038](adr/0038-operating-model-onboarding-agent-tax-and-incident-to-claim.md);
the falsifiable-claim machinery itself is the manifest in
[`scripts/lib/manifests/claims.ts`](../scripts/lib/manifests/claims.ts), driven by `pnpm prove` and gated
by `pnpm claims:verify`.

## The bar: not every postmortem earns a claim

A finding becomes a Tier-1 claim **only** if it is all three:

- **Load-bearing** — the property's silent failure actually matters (a tenant leak, a dropped delivery, a
  silent backlog), not a cosmetic nit.
- **Falsifiable** — you can name a single toggle that *makes the bug land*, and a gate that *reds* when it
  does. If you cannot make it red on demand, you do not yet understand the failure well enough to claim it.
- **Source-derived** — the gate asserts against a *single source*, not a hand-typed number. The strongest
  claims pin behaviour to a contract/constant/SLO, so the check cannot rot into theater.

If a finding is not all three, write it up in [`failure-modes.md`](failure-modes.md) instead — honest
documentation beats a weak claim. **Do not** manufacture a claim to grow the count; a synthetic claim is
the exact theater this discipline removes ([ADR-0030](adr/0030-checking-architecture-in-service-of-a-testing-mission.md)).

## The five steps

1. **Name the property the incident violated.** One sentence, in the claim grammar: *"&lt;claim&gt;. Breaks
   when &lt;toggle&gt;. Caught by &lt;gate&gt;."*
2. **Find or derive the single source** the gate will assert against (a Zod contract, a constant, an SLO, a
   spec).
3. **Add the deliberate-bug toggle** to the real (non-test) service code — the env var that *reintroduces*
   the failure when set. `pnpm claims:verify` checks a real service reads it (no orphan toggles).
4. **Write the gate** — the guarantee test that holds normally and reds with the toggle set — and add the
   manifest entry in [`claims.ts`](../scripts/lib/manifests/claims.ts) (id, boundary, technique, toggle,
   gate, evidence).
5. **Prove the teeth.** `pnpm prove <id> --break` must turn the gate RED. `pnpm claims:verify` re-runs that
   for every claim, so a claim that cannot be falsified cannot ship.

## Worked example: a stalled consumer → `consumer-lag-bounded`

This is a real claim in the manifest, born from a real-shaped incident class — **a durable JetStream
consumer falls behind under a burst (the moderator under load, the webhooks fan-out behind a slow
receiver), and the backlog grows silently with no defined failure mode.** Walked end to end:

- **The property** (step 1): *"A durable JetStream consumer keeps its lag within the consumer-lag SLO;
  stall it and `num_pending` + the oldest-unacked age climb past the bound, so the backpressure gate
  reds."* — the `consumer-lag-bounded` entry in [`claims.ts`](../scripts/lib/manifests/claims.ts).
- **The single source** (step 2): `CONSUMER_LAG_SLO` in
  [`packages/contracts/src/slos.ts`](../packages/contracts/src/slos.ts). The in-process gate
  (`evaluateConsumerLag`) **and** the Prometheus alert thresholds
  ([`deploy/observability/alerts.gen.yaml`](../deploy/observability/alerts.gen.yaml), generated by
  `pnpm alerts:gen`) both derive from it — one bound, two projections
  ([ADR-0034](adr/0034-observability-hardening-pii-free-spans-consumer-lag-slo-and-alert-rule-testing.md)).
  That is why the lesson is *tested SLO = alerted SLO* (see [reward-model.md](reward-model.md)): a stalled
  consumer that reds the gate also breaches the alert you are paged on.
- **The toggle** (step 3): `CHAOS_CONSUMER_STALL`, read by the messaging layer.
- **The gate** (step 4): the `@qaroom/messaging` property test "keeps consumer lag within the SLO".
- **The teeth** (step 5): `pnpm prove consumer-lag-bounded --break` arms the stall and the gate reds — the
  silent backlog is now a *defined, caught* failure mode instead of an incident waiting to recur.

No new claim was invented to write this page: the ritual is shown against a claim that already exists and
already falsifies, which is the honest demonstration the bar above demands.

# ADR-0001 — Foundational architectural decisions

**Status:** Accepted
**Date:** 2026-05-28
**Authored by:** Marius Argatu

## Context

QARoom is being designed as part of *The Testable Stack* series, a public exercise in designing microservices from a testing lens. Before any code is written, the architecture needs a set of foundational commitments that:

- Lock the lens through which all subsequent decisions are made.
- Distinguish architectural-level decisions (frozen for v1) from implementation-level decisions (made per milestone in future ADRs).
- Make explicit what is in scope and what is out of scope.
- Establish the substrate for the testing techniques the series intends to demonstrate.
- Leave deliberate seams for future capabilities (LLM agents, MCP, agentic CI/CD) without building those capabilities in v1.

This ADR is the immutable record of the 17 commitments arrived at through the design conversation that preceded any code. Superseding any of these commitments requires a new ADR that explicitly references this one.

## Decision

The following 17 commitments are locked for the lifetime of v1.

### 1. Microservices on Kubernetes
QARoom is implemented as five core services (plus a Milestone 9 sixth) running on Kubernetes. Local development uses k3d with Tilt as the inner-loop tool. CI uses KinD for ephemeral environments. Helm charts per service.

### 2. TypeScript end-to-end for the core; Python permitted for LLM services
The five core services are TypeScript: Fastify for HTTP, Drizzle for database, Zod for schemas, Vitest for unit testing. Python is permitted for LLM-adjacent services only — currently just the Milestone 9 community moderator using LangGraph.

### 3. Schema-first contracts with triangulation
Zod schemas are the source of truth for current contract shape. OpenAPI YAML is generated from Zod and committed. `oasdiff` runs in CI to gate breaking changes. Pact files are an independent second source of truth authored by consumers. Frozen `*.vN.yaml` specs at release boundaries provide the third source. No contract artifact is ever generated by an opaque process such that a developer can update it without independent verification.

The same discipline applies to async contracts. **AsyncAPI YAML is generated from Zod and committed per service.** Drift is gated in CI by `@asyncapi/diff` (or a thin custom check if `@asyncapi/diff` proves insufficient — see Milestone 0 spike). **Every event schema is versioned**: `packages/contracts/events/<name>.ts` is the current shape; `packages/contracts/events/<name>.v{N}.ts` is frozen at release boundaries. Breaking event changes require a major-version bump and a consumer-compatibility test that asserts existing consumers still parse the frozen prior version. Pact message contracts are the second source.

### 4. Sync REST + async messaging hybrid
REST is used for queries and external-facing endpoints. NATS JetStream is used for cross-service state-change events. All HTTP mutations carry an `Idempotency-Key` header.

**Single-writer-per-resource** is enforced at the persistence layer by a two-tier mechanism:

- **Postgres advisory locks keyed on the resource ID** (`pg_advisory_xact_lock(hashtextextended(resource_id, 0))`) acquired at the start of every mutating transaction. The lock serializes concurrent writers to the same resource within a single Postgres instance.
- **Row-level `SELECT … FOR UPDATE`** on the resource row inside the same transaction. The row lock is the fallback if the resource is new (no row to lock yet, advisory lock alone suffices) and the load-bearing guard once the row exists.

Idempotency-Key replays are served from a per-service `idempotency_responses` table keyed by `(idempotency_key, route, body_hash)` storing the original response. A second request with the same key returns the stored response, never re-executes. The table is housed in the service's own Postgres database; the schema is provided by `@qaroom/messaging` (alongside `processed_events`) as a reusable migration fragment.

Async at-least-once delivery is dedup'd as specified in Commitment 17.

### 5. All stateful flows are modeled as graphs
XState v5 for TypeScript flows. LangGraph for Python flows. Models live in `packages/contracts/` (or the Python equivalent), are hand-authored, and are the contract that production code and tests both consult. Production code queries the model to determine what is allowed; tests use the model to generate paths. State names are PascalCase and human-readable.

### 6. Determinism abstractions in every service
Every service accepts injectable `Clock`, `IdGenerator`, and `Randomness` interfaces. Production wires real implementations; tests wire seeded deterministic ones. Direct `new Date()`, `Math.random()`, or unseeded UUID generation in non-test code is a P0 defect, enforced by lint.

Two layers of time are explicit:
- **Logical time** flows through the injected `Clock`. Business logic (TTLs, expiries, state-machine timers) reads `clock.now()` only. Tests pin and advance this clock.
- **OS wall-clock time** is read by the runtime and by Chaos Mesh's `TimeChaos`. Chaos experiments perturb OS time.

Production code never observes OS wall-clock time directly. Chaos reproducibility is achieved by recording the **chaos configuration** (the TimeChaos manifest itself: target pods, skew direction, magnitude, duration) into the snapshot artifact (see Commitment 8) rather than the measured drift. Replays reapply the same manifest against the same cluster topology. The simplification is deliberate: detecting kernel-level `clock_gettime` interception from inside a process requires a trusted reference clock we do not have, and the per-experiment manifest is sufficient to make a chaos run replayable.

### 7. Observable state per service
Every service exposes:
- `GET /system/state` — current state of every model the service runs, in structured JSON. Every response includes an `as_of: {snapshot_id: string, lamport: number, wall_clock: ISO8601}` envelope. `snapshot_id` is a service-local opaque identifier for the read; `lamport` is a monotonic counter incremented on every write to any tracked model (DB-backed *or* in-memory XState actor). All mutating paths funnel through a single `LamportGate` in each service that bumps the counter and emits the new value in OTel attributes; XState transitions go through the same gate. Tests and MBT assertions pin reads against `snapshot_id` to avoid races with concurrent transitions.
- `GET /system/capabilities` — operations the service exposes, in MCP-tool-shaped JSON Schema form.

State reads are taken at Postgres `REPEATABLE READ` isolation so a single read returns a consistent snapshot across multiple models the service exposes. These endpoints are the substrate for both model-based testing and future MCP integration.

**Reverse conformance** (the system never enters a state outside the model) is enforced as follows: every XState actor wraps its `subscribe()` and emits an OTel span on every transition (`xstate.transition` span with `from`, `to`, `event` attributes). Tracetest assertions in CI verify that for every observed `xstate.transition`, both `from` and `to` are in the model's `states` set and `(from, event, to)` is a valid transition. Drift fails the assertion at trace-test time, not at model-conformance time.

### 8. Scoped scenario replay
Per-service `GET /system/snapshot` and `POST /system/snapshot` endpoints support capture and restore of database state plus observable state plus the current clock value. A `qaroom-replay` CLI orchestrates capture across services and reload into a Docker Compose environment. Documented limits: no in-flight HTTP request capture, no JetStream stream restore, no WebSocket session state.

This is deliberately less than full Antithesis-style deterministic simulation testing. The limits are part of the deliverable.

### 9. Communities are tenants
Each community is an isolated tenant. Tenancy is implemented as a shared schema with a `community_id` discriminator column. Cross-community data leakage is impossible by enforcement at the service layer and verified by property-based isolation tests (fast-check). Every span in OpenTelemetry carries `tenant.id`.

### 10. Donations are the first feature gated by a per-community state machine
The donation rollout state machine has explicit states (`DonationsOff`, `DonationsEnabling`, `DonationsOn`, `DonationsDisabling`) with documented observable behavior in each state. Per-community feature flags drive transitions. The state machine is the demonstration vehicle for model-based testing in Milestone 5.

### 11. WebSockets for server push, with a polling fallback
Real-time is a light commitment: notifications and live feed updates only, not collaborative real-time. Every WebSocket-delivered event is also retrievable via a polling endpoint, so agents and consumers without WebSocket support can consume the same data.

### 12. OpenTelemetry across all services, with GenAI semantic conventions opted in
Manual trace propagation through NATS message headers via a shared `@qaroom/messaging` SDK. Every span carries `tenant.id`. Errors carry RFC 7807 attributes on the span. The GenAI semantic conventions are opted in via `OTEL_SEMCONV_STABILITY_OPT_IN=gen_ai_latest_experimental` from Milestone 3 onward, even before any LLM calls exist.

### 13. All errors follow RFC 7807 Problem Details, extended for agents
Every non-2xx response uses `application/problem+json`. Three required extensions to the base envelope:
- `retryable: boolean`
- `next_actions: Array<{verb, href, description}>`
- `failure_domain: <enum>`

Failure domains are a closed enum maintained in `packages/contracts/errors.ts`. Schemathesis tests enforce the envelope in CI.

### 14. Test outputs are machine-readable
Every test runner in CI emits structured JSON or JUnit XML. A `test-results/summary.json` artifact with a frozen schema aggregates all results per PR. This schema is the contract that future agentic CI consumes. Breaking the schema is forbidden.

### 15. The substrate is agent-hospitable from day one
Required filesystem affordances, present from Milestone 0:
- `/AGENTS.md` and `/CLAUDE.md` at the repo root (CLAUDE.md is a symlink unless they diverge).
- Per-service `services/<name>/AGENTS.md`.
- `/.well-known/llms.txt` at the repo root and per service.
- `.claude/agents/` and `.claude/skills/` directories (initially empty with READMEs explaining the convention).
- `scripts/spin-up-ephemeral.sh` for provisioning namespaced ephemeral environments.

No agentic features are built in v1. The substrate exists so they can be added without rework.

### 16. The repo is a pnpm + Turborepo monorepo
One commit can change a service, its contracts, its consumers' contracts, and its tests atomically. Shared code lives in `packages/`. Services live in `services/`. Services may depend on packages; packages may not depend on services.

### 17. At-least-once async with explicit dedup
NATS JetStream is at-least-once. The architecture commits to the following dedup discipline so duplicate delivery cannot produce observable double-effects:

- **Publisher side.** Every publish sets the `Nats-Msg-Id` header to a deterministic event ID (`evt_<ulid>` from the injected `IdGenerator`). Streams are configured with `duplicate_window: 5m`. JetStream drops same-`Msg-Id` republishes within the window.
- **Publish atomicity.** Mutations that publish events use the transactional outbox pattern: the event row is written in the same Postgres transaction as the business write; a relay process publishes outbox rows to JetStream with the row's stable event ID, retried on failure until ack.
- **Consumer side.** Every consumer handler is idempotent. Either the handler is pure (state mutation derived from event content only), or the handler records the processed `event_id` in a per-subscription `processed_events` table inside the same Postgres transaction as its effects, and skips events it has already processed.
- **Cross-window safety.** Dedup beyond the 5m window is the consumer's responsibility (the `processed_events` table). The JetStream window is an optimization, not the contract.

This is verified in CI by a fast-check property test that fires duplicate events at handlers and asserts no double-effects, and by a Pact-message test that asserts the publisher sets `Nats-Msg-Id` on every emit.

## Consequences

### Positive

- The architecture is *sized exactly* for v1: nothing more than needed to demonstrate the testing techniques, nothing less than needed to make those demonstrations credible.
- Every architectural decision points back to one or more testing techniques it enables. The "why does this exist" question has a published answer for everything.
- The substrate for future capabilities (agents, MCP, agentic CI/CD, webhooks, continuous testing in production) is in place. None of these capabilities are built in v1, but adding any of them later is a milestone, not a rewrite.
- Triangulation defends against the auto-generated-artifact tautology that silently breaks most test suites.

### Negative

- The architecture is more complex than the product would justify in isolation. Five services for a product that could be a monolith is intentional; the cost is acknowledged.
- Bidirectional verification (Zod, OpenAPI, Pact, Schemathesis, frozen specs) means that contract changes touch multiple artifacts. The discipline is real and is documented in `docs/05-conventions.md`.
- The Kubernetes commitment slows down local development relative to Docker Compose. Tilt mitigates this; the cost is acknowledged.
- Milestone 7 (scoped scenario replay) is research-grade work and may not deliver on its full ambition in a single week. The mitigation (scoping the deliverable to documented limits) is part of the commitment.

### Trade-offs accepted

- **Operational overhead vs silent drift.** Bidirectional verification costs developer time on every contract change. We accept this cost because silent drift destroys the value of having tests.
- **Polyglot stack vs simplicity.** Adding Python for the LLM service complicates CI and tooling. We accept this because LangGraph + Python is the right tool for that milestone and forcing TypeScript would be performative.
- **Architectural complexity vs feature simplicity.** The product is deliberately modest while the architecture is rich. We accept this because the project's purpose is to teach architecture, not to ship features.

## Rejected alternatives

Each of these was considered and rejected, with the reason recorded:

- **Monolith with bounded contexts.** Rejected because microservices boundaries are the natural surface for contract testing, chaos engineering, and several other techniques. A monolith would force most techniques into unnatural shapes.
- **Docker Compose end-to-end.** Rejected because Chaos Mesh and the OpenTelemetry ecosystem are Kubernetes-native, and a hybrid setup would muddy the demonstrations.
- **gRPC for inter-service communication.** Rejected because Pact's REST support is more mature than its gRPC support, and Schemathesis works on OpenAPI. The performance cost of REST is irrelevant to a teaching project.
- **GraphQL.** Rejected because federated contract testing is immature and would not earn its place against the techniques the series already demonstrates.
- **Full Antithesis-style deterministic simulation testing.** Rejected as out of scope. The scoped scenario replay (Commitment 8) captures the principles without the multi-year engineering investment.
- **A service mesh** (Istio, Linkerd). Rejected because the testing story is rich enough without it; adding a service mesh would bury the demonstrations under wiring.
- **Real OAuth, real payments, multi-region, internationalization.** Rejected as deliberate exclusions to keep the architectural lens sharp.
- **Per-agent k3d clusters.** Rejected in favor of per-agent namespaces in a single cluster. The per-cluster approach is over-engineering for the demonstration purpose.
- **MCP servers per service in v1.** Rejected as designed-for-later. The `/system/capabilities` endpoint is the seam.
- **Generated-only Zod-to-OpenAPI without committed YAML.** Rejected as the auto-update tautology that breaks contract testing. The committed YAML and `oasdiff` are the defense.
